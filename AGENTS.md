# Repository Guidelines

## Project Structure & Module Organization

- `src/`: Fastify Papr plugin implementation (`fastify-papr-plugin.ts`), validation helpers (`mongo-validation-error.ts`), and barrel exports (`index.ts`).
- `tests/`: Vitest suites in TypeScript (`*.test.ts`) plus shared fixtures in `tests/helpers/`.
- `dist/`: Build artifacts emitted by `vp pack` (ESM + declarations); do not edit manually.
- `coverage/`: Generated by `vp test --coverage`; clean before commits if not needed.
- Root configs: `tsconfig.json` and `vite.config.ts` steer typing, packaging, linting, formatting, and tests.

## Build, Test, and Development Commands

- `pnpm install`: Install dependencies (Node >=22.12.0, pnpm 10.24+).
- `pnpm build`: Bundle to `dist/` with `vp pack` (ESM + dts).
- `pnpm test`: Run the Vitest suite through `vp test`.
- `pnpm lint`: Run `vp check` for formatting, linting, and type-checking; `pnpm lint:fix` applies `vp check --fix`.
- `pnpm fmt`: Auto-format via `vp fmt . --write` (TypeScript/JS/JSON/Markdown).
- `pnpm coverage`: Produce coverage reports via `vp test --coverage`; `pnpm make-badges` refreshes README badges.
- MongoDB driver requirement: v7+ (enforced at plugin initialization); `peerDependencies` declare the same.

## Coding Style & Naming Conventions

- TypeScript + ESM modules; prefer type-only imports/exports where possible.
- Formatting: Oxfmt via `vite.config.ts` (`printWidth: 120`, single quotes, no semicolons), 2-space indents; run `pnpm fmt` before pushing.
- Linting: Oxlint via the `lint` block in `vite.config.ts`; unused params should be prefixed with `_` to avoid warnings; `console` is only allowed in tests.
- Naming: follow descriptive camelCase for variables/functions, PascalCase for types/interfaces, and consistent `asCollection` model names matching Mongo collections.

## Testing Guidelines

- Tests live in `tests/*.test.ts`; use descriptive names mirroring feature or module (e.g., `validation.test.ts`).
- Integration tests rely on `mongodb-memory-server`; avoid hard-coding real connection strings.
- Add assertions for both success and failure paths (notably validation errors surfaced by `MongoValidationError` helpers).
- Aim to maintain the high coverage reflected in badges; update or add tests when changing behavior or types.

## Commit & Pull Request Guidelines

- Commit history follows Conventional Commit prefixes (`chore:`, `fix:`, `feat:`, etc.) with short imperative subjects.
- Before opening a PR, run `pnpm lint` and `pnpm test`; mention results in the description.
- PRs should summarize scope, link related issues, and include docs or changelog notes when behavior changes; add screenshots only if UI/docs output is affected.
- Keep diffs focused; refactors and feature work should land separately when practical.

## Security & Configuration Tips

- Do not commit secrets or real Mongo URIs; tests already use ephemeral in-memory servers.
- When adding new configuration knobs, document defaults and failure modes in `README.md` and keep runtime errors explicit (`fastify.log` is available).
